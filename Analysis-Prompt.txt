  COLLECT-EVIDENCE.PS1 — ANALYSIS AND REPORT GENERATION INSTRUCTIONS
  For use with GitHub Copilot CLI or any capable AI assistant
==========================================================================

HOW TO USE THIS FILE
--------------------
When you begin a new AI session to analyze evidence from Collect-Evidence.ps1,
paste or feed this file to the AI first, then say:

  "I have an evidence folder from Collect-Evidence.ps1 at [path].
   Generate a security audit report following the instructions in the
   prompt file I just provided."

The AI will read the evidence files and produce a consistent, standardized
report matching the format defined below.

For investigation reports (specific user behavior), say:
  "Also generate an investigation findings report for user [username]
   following the investigation report format in the prompt file."

==========================================================================
  PART 2: HOW TO ANALYZE THE EVIDENCE
==========================================================================

STEP 1 -- ORIENT (2 minutes)
  Read quick_indicators.json and cis_ig1_gaps.json first.
  These give you the machine-generated flags before you read anything else.
  Note: FirewallDisabledProfiles, RDPEnabled, RemoteAccessSoftwareHits,
  DormantAccounts, DaysSinceLastPatch.

STEP 2 -- SYSTEM BASELINE
  Read 00_metadata.txt and 01_systeminfo.txt.
  Note: OS version and build, domain membership (workgroup vs. AD),
  last boot time, hotfix count, RAM/processor (for context on machine role).

STEP 3 -- ACCOUNT REVIEW
  Read 01_local_users.txt. Note any accounts with:
  - Old or null LastLogon (dormant)
  - No password set date
  - Unusual account names

  Read 01_group_members_admins.txt. Anyone in Administrators besides
  the expected IT account and the built-in Administrator is a finding.

  Read 15_cis_accounts_all.csv, 15_cis_dormant_FLAGGED.csv,
  15_cis_password_noexpire_FLAGGED.csv.

STEP 4 -- SECURITY POSTURE
  Read 07_firewall_profiles.txt. Any profile (Private or Public) disabled
  is a finding. Domain profile disabled on a non-domain machine is lower risk.

  Read 07_rdp_settings.json. RDP enabled with no NLA = higher risk.
  RDP enabled on a machine with firewall disabled = compound risk (CRITICAL).

  Read 07_defender_status.json. Check: realtime enabled, behavior monitor,
  defs age vs. today, last full scan (never = finding regardless of realtime).

  Read 07_defender_exclusions.json. Any exclusion paths pointing to Temp,
  AppData, or user-writable directories = potential attacker abuse (HIGH).

  Read 18_cis_secure_config.json. SMBv1 enabled = CRITICAL.
  LLMNR or NetBIOS enabled = MEDIUM-HIGH (credential relay attacks).
  SMB signing not required = MEDIUM (relay attacks). LSA protection
  (RunAsPPL) not enabled = MEDIUM (credential theft).

  Read 04_services_localsystem_LOB.txt if it exists. Non-Microsoft services
  running as LocalSystem = LOW-MEDIUM (excessive privilege, common in SMB).
  Note: many LOB apps (QuickBooks, POS) legitimately require SYSTEM. Flag
  but use "commonly runs as SYSTEM" caveat when appropriate.

  Read 18_cis_screen_lock.json. No screen lock = MEDIUM.
  Timeout > 15 min = LOW. No password on resume = MEDIUM.

STEP 5 -- PATCH STATUS AND AUDIT LOGGING
  Read 16_cis_installed_hotfixes.csv (sort by InstalledOn descending).
  Last patch > 30 days = MEDIUM. > 90 days = HIGH. > 180 days = CRITICAL.
  Read 16_cis_pending_updates_FLAGGED.csv if it exists. Count Critical/Important.

  Read 18_cis_audit_log_sizes.json. Security log < 64MB = finding.
  Other logs < 32MB = finding. Note retention mode (Circular vs Archive).

  Read 18_cis_ps_logging.json. Script Block Logging disabled = MEDIUM.
  Module Logging disabled = LOW. Both disabled = MEDIUM-HIGH (no PowerShell
  forensic trail for incident response).

STEP 6 -- REMOTE ACCESS AND PERSISTENCE
  Read 03_processes_FLAGGED.txt -- anything here is HIGH priority.
  Read 04_services_SUSPICIOUS.txt -- services from non-standard paths.
  Read 05_scheduled_tasks_SUSPICIOUS.txt -- tasks running scripts.
  Read 06_autoruns_registry.txt -- look for non-Microsoft entries running
  from AppData, Temp, or user-writable locations.
  Read 08_installed_software.txt -- note: AnyDesk, TeamViewer, ScreenConnect,
  FileZilla, WinSCP, any FTP or remote access client.

  Read 13b_prefetch_FLAGGED.csv -- if this file exists, it means known
  attack tools were executed on this machine at some point (HIGH-CRITICAL).
  Even if the tools have since been deleted, Prefetch proves execution.

  Read 13c_wmi_persistence.csv -- any WMI event subscriptions are suspicious
  on workstations. ActiveScriptEventConsumer or CommandLineEventConsumer
  entries are almost always malicious (HIGH-CRITICAL).

  Read 13_registry_persistence.txt -- check Winlogon Shell/Userinit for
  unexpected entries, BootExecute for non-standard values, IFEO debugger
  entries (used for persistence and defense evasion).

STEP 7 -- NETWORK STATE AND EXTERNAL EXPOSURE
  Read 00_metadata.txt for the public IP address. Note this for potential
  external port scanning (Shodan, nmap) to assess internet-facing exposure.

  Read 07_dns_cache.txt -- look for unexpected external domains, cloud storage
  (mega.nz, gofile.io, anonfiles, FTP services), VPN providers, or C2 patterns.
  Read 07_netstat_connections.txt -- established connections to non-standard
  external IPs during the collection.
  Read 07_listening_ports.txt -- any process listening on 0.0.0.0 that isn't
  a recognized system service is a finding.
  Read 07_bits_jobs.txt -- upload-type BITS jobs to non-Microsoft destinations
  are potential exfiltration (HIGH).

STEP 8 -- USER ACTIVITY (for targeted investigations)
  Browser history files require binary mining (see SECTION 09 above).
  Key things to search for in Chrome History binary:
  - Financial system URLs (bank, ACH, accounting software)
  - Cloud storage upload URLs (drive.google.com upload, onedrive upload,
    dropbox upload, mega.co.nz, idrive.com)
  - Remote display / remote access downloads
  - Multiple Gmail accounts (u/0, u/1 pattern in URLs identifies each account)
  - Contacts export URLs (contacts.google.com/u/N/export or /label/)
  - File download records include: local save path, source URL, HTTP date header

  Read large_files_over50MB.csv -- sort by CreationTime. Bulk creation of
  large files (PST, ZIP, RAR) at the same timestamp = likely bulk copy operation.

  Read 02_logon_events.csv -- filter to specific user and date of interest.
  LogonType 10 = RDP. SourceIP for remote logons. After-hours logons are notable.

STEP 9 -- SHARES AND DATA EXPOSURE
  Read 17_cis_shares_all.csv and 17_cis_shares_OVERPERMISSIVE_FLAGGED.csv.
  Non-admin shares readable by Everyone = MEDIUM-HIGH depending on contents.


==========================================================================
  PART 3: SECURITY AUDIT REPORT FORMAT
==========================================================================

Generate the security audit report in this exact format. The report covers
infrastructure/configuration findings only -- no individual user accusations.
Save the final report as .md (source of truth) and .docx (via Pandoc + Word COM post-processing).

--------------------------------------------------------------------------
REPORT STRUCTURE:
--------------------------------------------------------------------------

1. EXECUTIVE SUMMARY (3-5 sentences)
   Brief statement of scope, machines examined, overall risk posture,
   and top priority. Written for a non-technical owner.

2. SCOPE AND METHODOLOGY
   - Systems examined (hostnames, OS versions)
   - Collection date
   - Tools used
   - Limitations (e.g., log retention, offline machines)

3. FINDINGS
   Each finding formatted as:

   3.X  [SEVERITY]  Finding Title
   CIS Control: [Control number and name]

   Description:
   [What was found, on which machine(s), specific technical detail]

   Risk:
   [What could happen as a result -- business impact language]

   Recommendation:
   [Specific, actionable remediation steps]

   Severity levels: CRITICAL / HIGH / MEDIUM / LOW / INFORMATIONAL

4. CIS IG1 COMPLIANCE SUMMARY
   Generate a comprehensive CIS IG1 compliance table from cis_ig1_gaps.json
   and the evidence files. The table must cover EVERY CIS IG1 safeguard
   that the script checks, organized by CIS Control number.

   Table columns: CIS Control | Safeguard | Status | Evidence File | Finding Ref
   Status values: PASS / FAIL / PARTIAL / NOT ASSESSED

   Group findings by CIS Control number so the report reads as a CIS IG1 audit.
   For controls marked NOT ASSESSED, note whether they require manual review
   (network-level controls, policy reviews, training records) and reference
   the CIS-IG1-Manual-Checklist.txt companion document.

   At minimum, the following controls MUST be individually assessed:
   CIS 1.1, 2.1, 3.3, 3.6, 4.1, 4.3, 4.4/4.5, 4.7,
   5.1, 5.2, 5.3, 5.4, 7.3, 7.4, 8.2, 8.3, 8.4, 8.5,
   9.1, 10.1, 10.2, 10.3, 10.4, 10.7, 12.2

5. REGULATORY CONTEXT
   For public water utilities: reference AWIA 2018, EPA cybersecurity
   guidance, and CISA water sector advisories. Note that CIS IG1 is
   the recommended baseline for water sector under EPA guidance.

6. PRIORITIZED REMEDIATION ROADMAP
   Three phases:
   - Immediate (address within 30 days): CRITICAL findings
   - Near-term (address within 90 days): HIGH findings
   - Planned (address within 12 months): MEDIUM findings + MSP/M365 migration

--------------------------------------------------------------------------
FINDING SEVERITY CRITERIA:
--------------------------------------------------------------------------
CRITICAL: Can be exploited remotely with no authentication, or represents
          active evidence of compromise. Example: firewall disabled + RDP
          enabled + no NLA = unauthenticated remote code execution path.

HIGH: Significant exposure requiring authentication or physical access to
      exploit, or represents a serious compliance failure. Example:
      shared/generic accounts, no MFA on financial systems.

MEDIUM: Increases attack surface or complicates incident response but
        requires chaining with other vulnerabilities. Example: password
        never expires, no full AV scan history.

LOW: Best practice deviation with limited direct exploitability.
     Example: Windows Search indexing sensitive file types.

INFORMATIONAL: Observations that inform remediation planning but are not
               vulnerabilities themselves.

--------------------------------------------------------------------------
TONE AND STYLE:
--------------------------------------------------------------------------
- Do NOT write in AI-typical language ("It is worth noting that...",
  "It's important to...", "In conclusion...", "Notably...")
- Write in direct, professional technical report language
- Use passive construction for findings: "RDP is enabled" not "You left RDP on"
- Findings should stand alone -- a reader unfamiliar with the client should
  understand each finding without reading others
- Avoid Unicode symbols (no checkmarks, no arrows, no bullets beyond hyphens)
- Do not pad findings with generic security advice -- be specific to what
  was found on these specific machines
- Remove all markdown formatting from the Word version




--------------------------------------------------------------------------
DOCUMENT GENERATION:
--------------------------------------------------------------------------

Use Pandoc to convert the markdown report to .docx. Do NOT use custom
PowerShell Word COM scripts for the conversion itself -- they are fragile
on complex/large documents. Use Word COM only for the post-processing pass.

Install Pandoc once (if not present):
  winget install --id JohnMacFarlane.Pandoc

In a new PowerShell session after install, refresh PATH first:
  $env:Path = [System.Environment]::GetEnvironmentVariable("Path","Machine") + ";" +
              [System.Environment]::GetEnvironmentVariable("Path","User")

Conversion command:
  pandoc "report.md" --from markdown --to docx `
    --reference-doc "reference-style.docx" --toc --toc-depth=3 `
    --output "report.docx"

The --reference-doc is a .docx with pre-configured heading and body styles.
Store a clean style template alongside this toolkit. Generate by running Pandoc
on minimal markdown, adjust styles in Word, save as the template.

Alternatively, use convert-report.ps1 (included in this toolkit):
  .\convert-report.ps1 -InputMd ".\report.md" -ReferenceDoc ".\reference-style.docx"

WORD COM POST-PROCESSING (run after every Pandoc conversion):
Applies table borders, heading size/spacing/AllCaps, and gray italic blockquote
styling. The Pandoc output alone does not include these formatting elements.

  $word = New-Object -ComObject Word.Application
  $word.Visible = $false
  $doc = $word.Documents.Open("C:\full\path\to\report.docx")

  foreach ($tbl in $doc.Tables) {
      $tbl.Borders.InsideLineStyle = 1
      $tbl.Borders.OutsideLineStyle = 1
      $tbl.Borders.OutsideLineWidth = 4
      $tbl.Borders.InsideLineWidth = 2
  }

  foreach ($para in $doc.Paragraphs) {
      switch -Wildcard ($para.Style.NameLocal) {
          "Heading 1" {
              $para.Format.SpaceBefore = 30; $para.Format.SpaceAfter = 8
              $para.Range.Font.Size = 22; $para.Range.Font.AllCaps = $true }
          "Heading 2" {
              $para.Format.SpaceBefore = 28; $para.Format.SpaceAfter = 6
              $para.Range.Font.Size = 16; $para.Range.Font.AllCaps = $true }
          "Heading 3" {
              $para.Format.SpaceBefore = 16; $para.Format.SpaceAfter = 4
              $para.Range.Font.Size = 13 }
          "Heading 4" { $para.Format.SpaceBefore = 10; $para.Format.SpaceAfter = 3 }
          "*Block Text*" {
              $para.Range.Font.Color = 0xAAAAAA   # Light gray (R=G=B=170)
              $para.Range.Font.Size = 10; $para.Range.Font.Italic = $true }
          "*Quote*" {
              $para.Range.Font.Color = 0xAAAAAA
              $para.Range.Font.Size = 10; $para.Range.Font.Italic = $true }
      }
  }

  $doc.Save(); $doc.Close(); $word.Quit()
  [System.Runtime.InteropServices.Marshal]::ReleaseComObject($word) | Out-Null


--------------------------------------------------------------------------
FORMATTING RULES:
--------------------------------------------------------------------------

1. NO SECTION NUMBERS in headings. Use descriptive heading text only.
   WRONG: "4. CIS IG1 Compliance Assessment" or "4.1 Firewall Review"
   RIGHT:  "CIS IG1 Compliance Assessment" or "Firewall Review"
   Reason: CIS IG1 uses X.X control numbering; numeric section headings
   create visual confusion when both appear in the same document.

2. FINDINGS GRID: Sort rows by severity descending.
   CRITICAL first, then HIGH, then MEDIUM, then LOW/INFORMATIONAL.
   Do not interleave severities. Number rows F-1, F-2... in that order.

3. CIS IG1 COVERAGE -- all 56 IG1 safeguards must be accounted for:
   a) N/A controls (e.g., CIS Control 16 for orgs with no software dev):
      Add a "Controls Not Applicable" table at the top of the CIS section
      with control numbers and a one-sentence justification for each.
   b) In-scope controls not assessable from endpoint/scan data alone
      (backup verification, MFA config, training records, documented
      processes): add a footnote after the CIS summary table listing
      safeguard numbers and why each requires manual/documentation review.
   Goal: the report must not appear to have omitted or forgotten any controls.

4. DISCLAIMER / LEGAL NOTICE: Place at the very end of the document after
   all appendices. Use markdown blockquote syntax (> text) so it renders
   as gray italic text after Word COM post-processing.

5. NO HORIZONTAL RULES (---) between sections. They render as dated
   page-width lines in Word. Section separation comes from heading AllCaps,
   increased font size, and SpaceBefore/SpaceAfter paragraph spacing.
   Strip all standalone --- lines from the markdown before conversion.

6. TABLE OF CONTENTS: Use Pandoc --toc flag (auto-generated from headings).
   Do not write a manual TOC in the markdown -- it will diverge over time.

7. RECOMMENDED SECTION ORDER:
   - Executive Summary
   - Scope and Methodology
   - Critical Findings  (brief -- top 3-5 most severe items only)
   - Detailed Technical Findings  (by device/system)
   - User Account Activity Review  (if applicable)
   - Threat Sweep Results  (if applicable)
   - CIS IG1 Compliance Assessment
   - Prioritized Remediation Roadmap
   - Appendix
   - [Disclaimer / Legal Notice in blockquote at very end]

==========================================================================
  PART 4: INVESTIGATION REPORT FORMAT (user-specific findings)
==========================================================================

When generating a user-specific investigation report, use this format.
This report is separate from the security audit and is for the owner's
eyes only. It should be factual and evidence-cited -- no speculation.

IMPORTANT: Only assert what the evidence directly shows. If something
is suspicious but explainable, say so and note what would confirm or
refute it. Do not imply guilt without direct evidence.

--------------------------------------------------------------------------
INVESTIGATION REPORT STRUCTURE:
--------------------------------------------------------------------------

1. PURPOSE AND SCOPE
   Who authorized the investigation, which user, which machine,
   date range of evidence examined.

2. FINDINGS SUMMARY
   Brief table: Finding | Evidence | Confidence (Confirmed/Probable/Possible)

3. DETAILED FINDINGS
   Each finding:
   - What was observed (with specific file names and timestamps)
   - What it means
   - Alternative explanations if any
   - What additional evidence would confirm it

4. TIMELINE OF KEY EVENTS
   Chronological table of notable events with source file cited for each.

5. OPEN QUESTIONS
   List specific things that need verification (physical receipts, IT
   consultant confirmation, server-side logs, etc.)

6. RECOMMENDED NEXT STEPS
   What to do next -- second site visit items, server-side log reviews,
   HR/legal consultation triggers.

--------------------------------------------------------------------------
CONFIDENCE LEVELS:
--------------------------------------------------------------------------
CONFIRMED:  Direct, unambiguous evidence. No innocent explanation fits.
PROBABLE:   Strong evidence; innocent explanation exists but is unlikely
            given the combination of findings.
POSSIBLE:   Suspicious but explainable by normal business activity.
            Requires corroboration before drawing conclusions.
BENIGN:     Initially flagged, determined to have innocent explanation.

==========================================================================
  PART 6: COMMON IOC PATTERNS TO WATCH FOR
==========================================================================

These patterns in the evidence files warrant investigation:

REMOTE ACCESS TOOLS (03_processes_FLAGGED.txt, 08_installed_software.txt):
  AnyDesk, TeamViewer, ScreenConnect, ConnectWise Control, LogMeIn,
  GoToAssist, Ammyy Admin, UltraVNC, DWService, RustDesk
  -> Normal for IT support (one-time, customer-initiated)
  -> Suspicious if running as a service (persistent), or if user installed it

FTP/FILE TRANSFER TOOLS:
  FileZilla, WinSCP, CuteFTP, Total Commander FTP
  -> Check FileZilla recentservers.xml for destination servers
  -> Any external FTP destination = HIGH priority follow-up

CLOUD STORAGE EXFIL INDICATORS (07_dns_cache.txt, browser history):
  mega.nz, gofile.io, anonfiles.com, sendspace.com, zippyshare.com
  = unauthorized cloud drop sites (HIGH-CRITICAL)
  rclone = command-line cloud sync tool, often used for bulk exfil

DEFENDER EXCLUSION ABUSE (07_defender_exclusions.json):
  Any exclusion path pointing to Temp, AppData, Downloads, or user-writable
  directories = potential attacker abuse to hide malware (HIGH).
  Exclusion of entire drives (C:\) = CRITICAL.
  Any process exclusion for non-Microsoft executables = review needed.

PREFETCH ARTIFACTS (13b_prefetch_FLAGGED.csv):
  Prefetch proves an executable was run even if the file was later deleted.
  If attack tools (mimikatz, psexec, rclone, etc.) appear in Prefetch,
  the machine was compromised regardless of current running state. The
  LastWriteTime on the .pf file approximates the last execution time.

WMI PERSISTENCE (13c_wmi_persistence.csv):
  ActiveScriptEventConsumer or CommandLineEventConsumer entries on
  workstations are almost always malicious. WMI persistence survives
  reboots and is invisible to most users and many AV products.
  Any findings here warrant full incident response.

BROWSER HISTORY PATTERNS:
  Multiple signed-in Google accounts (u/0, u/1 in URLs) = normal but
  note which account accesses company resources vs. personal resources
  Contacts export URLs (contacts.google.com/.../export) = review what
  account the export came from and what label was exported
  Financial system URLs + download of financial docs = within role unless
  combined with other exfil indicators

PST FILE ANALYSIS:
  PST files for OTHER users' mailboxes on the target user's machine = HIGH
  Multiple PST files created at identical timestamps = bulk copy operation
  Large PST files (>500MB) combined with cloud backup tool = exfil risk


==========================================================================
  PART 7: GATHERING CLIENT CONTEXT
==========================================================================

Before beginning analysis, always ask the user for client-specific context.
This context directly affects how findings are classified (suspicious vs.
normal business activity) and what regulatory requirements apply.

Ask the following questions before reading any evidence files. If the user
has already provided some answers, skip those questions.

REQUIRED QUESTIONS:

1. "What type of business is this, and are they subject to any regulatory
   frameworks? (e.g., public utility, healthcare, financial services,
   government contractor)"

2. "What are the machine names or hostnames I will be analyzing, and what
   is each machine's role? (e.g., front desk, server, shop floor)"

3. "Is there a specific user whose behavior is the focus of this
   investigation, and what is their job role/responsibilities?"

4. "Who is the IT contact or MSP for this company, and when did they
   last have physical access to the machines?"

5. "What software is known to be legitimately installed? List any remote
   access tools, backup software, financial systems, or custom line-of-
   business applications that are expected."

6. "Are there any shared or generic accounts that are intentionally used
   by multiple people? (e.g., a timeclock login, a shared service account)"

7. "Is there a perimeter firewall in place, and if so, what make/model?"

OPTIONAL BUT USEFUL:

8. "Are there any events or dates of particular interest I should focus on?"

9. "Have there been any recent personnel changes, IT visits, or unusual
   incidents before this investigation was initiated?"

10. "What is the investigative goal -- general security audit, specific
    user investigation, or both?"

Once context is gathered, proceed with analysis following Parts 1-6.
Reference the client context throughout to distinguish normal business
activity from genuinely suspicious findings. Document which findings
required context to classify (this helps identify gaps in documentation
and policy at the client site).

==========================================================================
  PART 8: FACTUAL ACCURACY AND ANTI-HALLUCINATION RULES
==========================================================================

CRITICAL: These rules override all other instructions when in conflict.

RULE 1 -- EVIDENCE-ONLY ASSERTIONS
  Every factual claim in the report must be directly traceable to a
  specific evidence file, configuration line, or scan output. If a claim
  cannot be verified from the evidence, it MUST NOT appear in the report.

  NEVER infer, assume, guess, or fabricate:
  - Version numbers, build numbers, or dates not found in evidence
  - IP addresses, MAC addresses, or hostnames not in evidence
  - Account names, login dates, or group memberships not in evidence
  - Configuration values (enabled/disabled/value) not explicitly in config
  - Software install dates not in evidence files
  - Service states (running/stopped) not in evidence files

RULE 2 -- DEFAULTS AND ABSENCES
  When a configuration value is not explicitly set in a config file:
  - Do NOT state the value as if it were observed
  - Instead write: "[setting] not explicitly configured (FortiOS default: [value])"
  - If you are uncertain of the default, state: "[setting] not present in
    configuration; verify manually"
  - The absence of a setting is NOT the same as observing it disabled

RULE 3 -- WORDING FOR UNVERIFIABLE CLAIMS
  If a claim comes from client conversation or on-site observation rather
  than collected evidence files, use one of:
  - "Per client report, [claim]" (for verbal/email info from the client)
  - "Observed during on-site assessment: [claim]" (for in-person findings)
  - "Not present in collected evidence; requires manual verification"
  Do NOT present these as machine-verified findings.

RULE 4 -- PROHIBITED PATTERNS
  The following patterns indicate hallucination and must be avoided:
  - Stating a specific value from a file you have not read
  - Copying a value from one device and attributing it to another
  - Inventing remediation steps for features that are already configured
  - Reporting a feature as "disabled" when the config actually shows "enable"
  - Filling in placeholder data with plausible-sounding values
  - Stating install dates, event dates, or timestamps not in evidence

RULE 5 -- TWO-PASS WORKFLOW
  After generating the report, perform a second complete pass:
  1. Re-read EVERY factual claim in the report
  2. For each claim, identify the exact evidence file and location
  3. If a claim cannot be sourced, either remove it or add a caveat
  4. Generate the Citation Verification Table (see Part 9) as proof

RULE 6 -- ERROR ACKNOWLEDGMENT
  If you discover an error during the verification pass:
  - Fix the error immediately
  - Add a note in the Citation Verification Table documenting:
    a) What was originally stated
    b) What the evidence actually shows
    c) What was corrected to
  Do NOT leave known errors in the report under any circumstances.


==========================================================================
  PART 9: CITATION VERIFICATION TABLE (MANDATORY COMPANION DOCUMENT)
==========================================================================

Every report MUST be accompanied by a Citation Verification Table saved
as a separate file alongside the report. This is non-optional.

FILE NAMING:
  If the report is: ClientName-CIS-IG1-Security-Assessment.md
  The table is:     ClientName-Citation-Verification.md

PURPOSE:
  The citation table maps every factual claim in the report to its source
  evidence file, enabling independent verification. It also serves as a
  quality control artifact proving the report was fact-checked.

TABLE FORMAT:
  Each section of the report gets its own table with these columns:

  | # | Claim | Source File | Evidence | Status |
  |---|-------|-------------|----------|--------|

  Where:
  - #:          Unique claim identifier (e.g., F1-01, N-03, MO-15)
  - Claim:      The specific factual assertion from the report
  - Source File: The exact evidence filename (e.g., 01_local_users.txt)
  - Evidence:   The specific line, value, or entry that supports the claim
  - Status:     One of the following:
      VERIFIED   -- Claim matches evidence exactly
      CAVEAT     -- Claim is mostly correct but has a nuance (explain)
      INCORRECT  -- Claim does not match evidence (MUST be corrected)
      VERBAL     -- Claim comes from client/assessor, not evidence files
      UNVERIFIED -- No evidence file found to confirm or deny

SUMMARY TABLE:
  At the end, include a summary:

  | Category | Verified | Caveat | Incorrect | Verbal | Unverified | Total |
  |----------|----------|--------|-----------|--------|------------|-------|

  The report cannot be delivered if any claims have status = INCORRECT.
  All INCORRECT items must be fixed before delivery.

WORKFLOW:
  1. Generate the report following Parts 1-7
  2. Generate the citation table by re-reading each claim against evidence
  3. Fix any INCORRECT items found during step 2
  4. Re-verify the fixed items
  5. Deliver both the report and the citation table together

IMPORTANT: The citation table is a DELIVERABLE, not a draft artifact.
It should be professional, organized by report section, and complete.
If the table shows 100% VERIFIED with zero INCORRECT, it provides
confidence that the report is factually accurate. Any CAVEAT or
UNVERIFIED items should be reviewed by the analyst before delivery.
